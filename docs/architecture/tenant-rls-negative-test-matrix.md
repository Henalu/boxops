# Tenant / RLS Negative Test Matrix - BoxOps

Estado 2026-05-20. Esta matriz consolida las pruebas negativas de tenant, RLS y permisos que BoxOps debe tener por superficie critica antes de llamar beta interna o produccion a un entorno con datos reales. Es una fase documental local y prudente: no implementa tests, codigo, UI, migraciones, seeds ni features.

No desbloquea S.8/A.1, B.4, OD.1/I.32 ni F.15. Esos gates siguen bloqueados mientras falten acceso real/staging, URL QA/staging, Supabase project/ref o DB URL, credenciales E2E por rol, tenant QA/staging, datos operativos controlados, SMTP real y evidencia redacted.

Esta matriz deriva de `docs/architecture/asvs-level-1-beta-matrix.md`, `docs/architecture/security-baseline.md`, los runbooks de readiness beta y la revision local de helpers, Server Actions, rutas, migraciones, snippets y smokes existentes. El plan tecnico priorizado para convertirla en tests vive en `docs/architecture/tenant-rls-negative-test-implementation-plan.md`. No es auditoria profesional, pentest, certificacion ASVS, cumplimiento legal definitivo ni aprobacion de beta.

## Alcance Y Estados

La matriz usa estos estados:

| Estado | Significado |
|---|---|
| `cubierto-local` | Ya existe evidencia local/documental o smoke/snippet rollback que cubre el caso de forma razonable para este corte documental. No equivale a staging validado. |
| `parcial` | Hay controles implementados o documentados, pero faltan pruebas negativas unificadas, cobertura por rol, evidencia real o cierre de deuda. |
| `bloqueado-por-entorno` | No puede cerrarse desde este workspace sin acceso real/staging, credenciales, tenant, datos, SMTP, Storage controlado o evidencia externa. |
| `pendiente-test` | Debe convertirse en test negativo o snippet especifico en una task tecnica posterior. |
| `no-aplica-beta` | Queda explicitamente fuera de beta interna y no debe abrirse en este corte. |

Reglas de lectura:

- Los casos `bloqueado-por-entorno` no deben reintentarse sin acceso nuevo concreto.
- Los casos `pendiente-test` no autorizan a ampliar producto; solo definen deuda de verificacion.
- Los casos `no-aplica-beta` son guardrails de exclusion, no backlog encubierto.
- Toda evidencia real con datos, usuarios, documentos, URLs, cookies, signed URLs o secretos debe vivir fuera del repo y redacted.

## Decision
