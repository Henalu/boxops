# Tenant Direct Grants Runtime Scenario Manifest - BoxOps

Estado 2026-05-20. Este manifiesto aterriza el runbook `docs/operations/tenant-direct-grants-runtime-validation-runbook.md` en escenarios runtime trazables antes de aplicar el draft S.77. No ejecuta runtime, no aplica el draft, no crea migracion `00044`, no cambia grants reales, no cambia default privileges, no toca `src` y no desbloquea beta.

## Decision

S.79 aporta cobertura documental distinta de S.78 porque S.78 define capas, secuencia y criterios generales, mientras este manifiesto asigna cada tabla con DML directo observado a archivos/actions/helpers/rutas, rol minimo, caso feliz, negativos por rol, negativos cross-tenant y evidencia esperada.

El estado de `Direct SQL grants` sigue siendo `parcial`: esto no es evidencia de PostgREST, Server Actions, navegador, Storage, Auth/SMTP, staging ni beta.

## Reinspeccion Local 2026-05-20

Catalogo local revalidado antes de escribir:

- `pg_class.relacl`: 40 tablas publicas de app revisadas; 40 tienen RLS habilitado.
- `pg_policies`: 94 policies en `public`; 0 policies para `anon`; 94 policies para `authenticated`.
- `information_schema.role_table_grants`: `anon` conserva `SELECT/INSERT/UPDATE/DELETE/TRUNCATE/REFERENCES/TRIGGER` en 32 tablas; `authenticated` conserva `SELECT` en 40 tablas y `INSERT/UPDATE/DELETE/TRUNCATE/REFERENCES/TRIGGER` en 29 tablas.
- `pg_default_acl`: default table ACL de `postgres` y `supabase_admin` en `public` sigue concediendo `anon=arwdDxt` y `authenticated=arwdDxt`; queda como operacion separada de owner/operador.
- DML directo Supabase en `src`: siguen apareciendo 14 tablas con `.insert`, `.update`, `.upsert` o `.delete` reales desde app/helpers; los matches de `URLSearchParams.delete(...)`, `Set.delete(...)` y `createHash(...).update(...)` no cuentan como DML de Supabase.
- RPCs en `src`: siguen existiendo flujos `SECURITY DEFINER` que deben validarse por comportamiento, no solo por `EXECUTE`.
- Migraciones `GRANT`/`REVOKE`: las tablas MVP tempranas conservan grants amplios; varias tablas sensibles posteriores revocan DML directo y usan `SELECT`/RPC; el draft S.77 no reduce `authenticated` a `SELECT`.

## Pruebas Locales Posibles Vs Bloqueadas

- Repetir inventario de catalogo y verificador rollback de S.77.

- Ejecutar smokes existentes sin aplicar el draft.
- Ejecutar escenarios runtime solo si ya existen credenciales E2E locales sinteticas por rol y datos A/B controlados; si faltan, marcar `bloqueado`, no simular.

Bloqueado sin acceso nuevo real o harness dedicado:

- Server Actions con cookies/sesion real por `owner`, `admin`, `manager` y `coach`.

## RPCs Que Acompanian El Manifiesto

Estos grupos no sustituyen la tabla anterior. Deben probarse como comportamiento cuando exista harness, porque el draft S.77 no solo afecta DML directo sino tambien la confianza de los flujos que dependen de RPCs `SECURITY DEFINER` y grants `EXECUTE`:

- Invitaciones: `get_team_invitation_public(...)` y `accept_team_invitation(...)` con token/email correctos e incorrectos.
- Tipos: `update_class_type_and_sync_defaults(...)` y sincronizacion de defaults.
- Settings de fichaje: `update_organization_time_tracking_config(...)`.
- Mi cuenta: lifecycle RPC de avatar/firma propios; Storage real queda bloqueado.
- Cambios, ausencias, eventos, overtime, time-location inactivo, fichaje/cierre semanal, auditorias y document programming: probar solo con datos sinteticos o entorno autorizado.
- Documentos/archivo: preview/download E.5, auditoria backend real, objeto `document-files`, policies Storage y TTL siguen bloqueados hasta E.16/S.8 con acceso real.

## Criterio Para Usarlo Antes De Migracion Real

1. Ejecutar S.74-S.77 como preflight SQL/catalogo en el entorno candidato, sin llamarlo runtime.
2. Preparar fixtures A/B sinteticas y usuarios por rol o documentar bloqueo concreto.
3. Ejecutar los casos felices y negativos de este manifiesto antes y despues de aplicar el draft en un entorno desechable/autorizado.
4. Registrar pass/fail redacted por tabla, accion, rol y tenant; no guardar secretos, cookies, signed URLs, rutas Storage ni contenido documental.
5. Mantener `Direct SQL grants` en `parcial` hasta que exista migracion aplicada y validada en el entorno declarado.
6. Tratar default privileges de `postgres` y `supabase_admin` como task separada, no como parte de S.77/S.79.
